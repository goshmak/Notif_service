============
config.py
============
Централизованная конфигурация, загружаемая из переменных окружения через pydantic-settings.


====================
create_content.py
====================
Генерирует содержимое уведомления (тема, HTML-тело, текстовое тело, сообщение VK)
с использованием шаблонов Jinja2.

Шаблоны:
templates/<notification_type>_email.html - расширенный HTML для электронных писем
templates/<notification_type>_email.txt - резервный текстовый вариант для электронных писем
templates/<notification_type>_vk.txt - короткий текст для сообщений VK

ContentBuilder.build() возвращает словарь с ключами:
subject, body_html, body_text, vk_text


==============
database.py
==============
Асинхронный механизм базы данных и фабрика сессий.

Поддерживает как SQLite (прототип), так и PostgreSQL (финальная версия).
Переменная среды DATABASE_URL определяет, какая БД используется.


=========
main.py
=========
Точка входа для микросервиса модуля уведомлений.

Функции:
- Инициализация приложения FastAPI
- Менеджер жизненного цикла (хуки запуска/остановки для БД и Redis)
- Все определения HTTP-эндпоинтов
- Валидация запросов с помощью моделей Pydantic
- Немедленное подтверждение + фоновая обработка очереди


============
models.py
============
Уровень данных для модуля уведомлений.

Содержит:
- Перечисления для типов уведомлений, статусов и каналов
- Модели ORM SQLAlchemy (NotificationRecord, NotificationSettings)
- Схемы Pydantic для валидации запросов/ответов


=================
redis_queue.py
=================
Реализация очереди задач с использованием списков Redis (паттерн BLPOP/RPUSH).

Архитектура:
- Производители вызывают `enqueue_notification()` для отправки задач в формате JSON в список Redis (очередь).
- Рабочие процессы вызывают `run_worker()`, который зацикливается на BLPOP и обрабатывает задачи параллельно с использованием asyncio.
- Неудачные задачи повторно ставятся в очередь с экспоненциальной задержкой до MAX_RETRY_ATTEMPTS, после чего запись в базе данных помечается как FAILED.


=======================
send_notification.py
=======================
Уровень бизнес-логики: маршрутизация уведомления в правильные каналы доставки.

Функции:
1. Определить, какие каналы использовать (из запроса + настроек пользователя).
2. Вызвать соответствующий отправитель (EmailSender или VKSender).
3. Генерировать исключения при сбое доставки, чтобы очередь могла повторить попытку.

Порядок разрешения каналов:
- Если request.channel == "all" -> во все включенные и настроенные каналы.
- Если request.channel == "email" -> только email.
- Если request.channel == "vk" -> только VK.


===========
worker.py
===========
Автономная точка входа в процесс worker.
Запускается отдельно от сервера FastAPI для обработки очереди уведомлений:
Worker использует те же настройки config.py и Redis, что и main.py.
Он предназначен для запуска в качестве отдельного процесса или службы systemd.
Несколько экземпляров worker могут работать параллельно с одной и той же очередью Redis.
